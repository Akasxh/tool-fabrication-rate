## What it is
AppWorld Engine is a high-fidelity, in-process sandbox of **9 day-to-day apps** (Amazon, Spotify, Venmo, etc.) exposed via **457 local APIs** over ~100 DB tables, populated with the digital activities of ~100 simulated people. On top sits the **AppWorld Benchmark**: complex day-to-day agent tasks (train / dev / test_normal / test_challenge) where the agent must complete a supervisor's instruction by **writing interactive Python code that calls these APIs** (ReAct-style code agent), then `apis.supervisor.complete_task()`. Evaluation is **state-based** (final DB state via per-task evaluation programs), not trajectory/process-based. APIs run via FastAPI TestClient in the same process (no server needed; Docker optional). Ships an MCP server/client, a CLI (`appworld run/evaluate/serve`), and an `experiments` agents package supporting 5+ agents and "100+ models".

## Fit with our paper (TEHR / RVR on tool-existence hallucination)

Our metric (TEHR) is **per-call tool-existence hallucination**: model emits a tool name; we check membership against a fixed per-task **tool registry**, then RVR re-prompts with the registry on a miss. Our harness loaders (`harness/bench_loaders/bfcl.py`, `tau_bench.py`) yield `Task` objects carrying a canonical OpenAI-shape `registry` dict, and adapters do **native function-calling**.

Critical mismatches:
1. **Code-agent, not function-calling, by default.** AppWorld's canonical interface is free-form Python in an IPython shell (`world.execute(code)`), where "tool calls" are `apis.spotify.login(...)` expressions inside arbitrary code. A hallucinated API surfaces as a Python `AttributeError`/`NameError` at execution, not as a discrete out-of-registry tool-call event. TEHR as we define it (a clean per-call name-vs-registry check) does **not** map cleanly onto code execution — we'd have to parse API references out of arbitrary Python, which is brittle and not equivalent to our other benchmarks.
2. **Huge / open tool space.** 457 APIs across 11 apps, and the agent is *expected* to discover APIs at runtime via the `ApiDocs` app rather than be handed a fixed per-task list. There is no small, closed per-task registry like BFCL's `involved_classes` → tools. "Tool existence" is fuzzier: the universe is large and partly self-served.
3. **State-based eval, no per-call oracle.** Scoring is DB-state diff at the end, so it gives task success, not the per-call labels our stats stack (`stats/tehr.py`, bootstrap CIs, McNemar) consumes.

There **is** a function-calling on-ramp: `data/api_docs/function_calling/{app}.json` ships API docs in OpenAI function-calling format, and the repo includes a tool-use agent. So one *could* run AppWorld in FC mode and define the per-task registry as the union of FC schemas for the task's required apps. But that registry is still large/open and the eval remains state-based — it's a research project, not a drop-in.

## Concrete verdict per use

- **RUN as an extra benchmark — NO (not for the headline TEHR result), MAYBE as a stretch/secondary.** Effort is HIGH. It needs: install + LFS pull + `appworld install` (decrypt bundles) + `appworld download data`, then either (a) wrap the code-agent loop and invent a code-reference-extraction TEHR proxy, or (b) drive it in function-calling mode and synthesize a per-task registry from `api_docs/function_calling/`. Both diverge from how we score BFCL/τ-bench, hurting cross-benchmark comparability. For ICML breadth, BFCL + τ-bench + (a cleaner FC benchmark like ToolBench/API-Bank/NexusRaven-style) buys more than fighting AppWorld's code-agent paradigm. Treat AppWorld as a **breadth stretch goal only if** we add a code-agent variant of TEHR.
- **Reuse a COMPONENT in our harness — NO.** Our harness contracts (`HARNESS_SPEC.md`) are built around native function-calling adapters and a closed per-task registry. AppWorld's FastAPI-TestClient env, stateful IPython shell, and per-task DB evaluators don't slot into our `Callable[[str,dict],dict]` tool_executor cleanly the way BFCL's stateful class instances do. Pulling it in would be a parallel subsystem, not a component.
- **Reuse as a BASELINE — NO.** It's an environment/benchmark, not an intervention or a model we'd compare RVR against.
- **Cite as PRIOR ART — YES (recommended).** Strong, well-known citation for the related-work / benchmarks section: ACL'24 Best Resource Paper, frequently cited for interactive coding / function-calling agents. Use it to (a) motivate that tool/function-calling agents operate over large, partly-discovered API spaces, and (b) **differentiate**: AppWorld measures end-task success via state diff and uses a code-agent over an open API world, whereas we isolate a per-call tool-existence-hallucination failure mode with a closed registry and a targeted re-prompt intervention. Good contrast for "why a focused TEHR metric is needed beyond aggregate task success." Cite the arXiv 2407.18901 / ACL 2024 entry (bibtex in repo).
- **Borrow a PATTERN for the paper-revision skill / reviewer personas — MINOR.** Two transferable ideas: (1) the **canary string + encrypted-bundle contamination-control** practice is a concrete "benchmark hygiene" checklist item a reviewer persona could raise (does the paper guard against test-set leakage?); (2) the **train/dev/test_normal/test_challenge split with hidden setup+solution on test** is a reusable rubric point for evaluating contamination claims. Low priority but cheap to fold into a reviewer persona's checklist.

## License risk (be honest)
Apache-2.0 overall, **but split into two tiers**:
- **Public portion** (agent baselines, eval utils, execution shell, tutorials) — plain-text Apache-2.0, fully reusable.
- **Protected portion** (API docs/implementations, task data, solutions, eval tests) — shipped as **encrypted `.bundle` (git-LFS) files**, Apache-2.0 **plus an added clause that any public redistribution of it (or derivatives) must remain encrypted.** Plus a canary string and an explicit request not to post extracted code/data in plaintext online.
- **Implication for us**: Training and serving model outputs are explicitly *not* redistribution, so **running it for evaluation is fine.** Risk is: if we vendor/commit any decrypted task data, API docs, or derived artifacts into our (public) repo or paper artifacts, we'd breach the encryption-on-redistribution clause and the no-plaintext-posting request. Mitigation: keep AppWorld data outside the repo (it lives in `./data` / `APPWORLD_ROOT` by design), never commit decrypted bundles, don't paste task/API text into the paper. With that discipline, license risk is LOW-to-MEDIUM; the trap is accidental artifact leakage, not the eval itself.

## Bottom line
Cite-it. Don't run it as a core TEHR benchmark — paradigm mismatch (code-agent + open 457-API space + state-based eval) makes effort HIGH and comparability poor. Best value is as differentiated prior art and a contamination-hygiene pattern source.
